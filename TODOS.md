## V2: localStorage 进度持久化

**What:** 用 localStorage 保存 ProgressState，刷新页面后恢复进度。

**Why:** 用户在体验中途刷新页面会丢失进度，需要从头开始。5 分钟的体验流程中途被打断（来电话、误触返回）是常见场景。

**Pros:** 实现简单（~10 行代码），用户体验明显提升。

**Cons:** 需要处理数据版本兼容（场景数据结构变化时旧缓存可能不兼容）。

**Context:** useProgress.ts 的 ProgressState 已经定义好了，只需要在状态变化时写入 localStorage，初始化时读取。加一个版本号字段，数据结构变化时自动清除旧缓存。

**Depends on:** V1 完成

## V2: 场景贡献模板

**What:** 创建 scenario-template.ts 模板文件和填写指南，让社区贡献者只需要填数据就能添加新场景。

**Why:** 开源项目的生命力在于社区贡献。如果贡献新场景需要看懂整个代码库，门槛太高。

**Pros:** 降低贡献门槛，加速场景积累，让非程序员（比如心理咨询师、职业规划师）也能贡献。

**Cons:** 需要维护模板和指南的同步更新。

**Context:** Scenario 类型定义已经很清晰。模板文件只需要把类型定义变成一个带注释的空壳，加上每个字段的填写说明和示例。放在 CONTRIBUTING.md 里引导贡献者。

**Depends on:** V1 完成 + 至少 1 个场景验证通过

## V2: AI 个性化生成

**What:** 用户输入自己的真实纠结，AI 实时生成个性化的 4 条路线、时间线、分叉和收口。

**Why:** 预写场景只能覆盖高频题材，AI 生成能让每个用户都获得针对自己处境的分析。这是产品从"体验工具"变成"真正有用的决策辅助"的关键一步。

**Pros:** 极大扩展产品价值，每个用户的体验都是独一无二的。

**Cons:** 需要用户接自己的 API key（门槛高）；AI 生成质量不稳定需要 prompt 工程；增加前端复杂度（loading 状态、错误处理、流式输出）。

**Context:** 技术方案：前端直接调用 LLM API（无后端），用户在设置页输入 API key 存入 localStorage。Prompt 需要精心设计，确保输出符合 Scenario 类型结构。可以先支持 OpenAI 兼容 API（覆盖大多数国内外模型服务商）。

**Depends on:** V1 完成 + 预写场景验证通过 + Scenario 类型稳定

## V2: 正式设计系统文档（DESIGN.md）

**What:** 创建 DESIGN.md，把 V1 设计文档中的 Design Tokens（色板、字体、间距、动效规则）提取为独立的设计系统文档。

**Why:** V1 的视觉规范散落在设计文档里，后续加新场景或页面时，实现者需要翻设计文档找 token。独立的 DESIGN.md 能保证多场景、多贡献者的视觉一致性。

**Pros:** 贡献者和 AI 工具都能快速找到视觉规范，减少"每个场景长得不一样"的风险。

**Cons:** 需要维护两份文档的同步（设计文档 + DESIGN.md）。

**Context:** V1 已经定义了完整的 Design Tokens（深夜温暖色色板、system-ui 字体、间距系统、3 种动效规则）。只需要把这些提取到 DESIGN.md，加上使用示例和组件级别的规范。可以用 `/design-consultation` 生成。

**Depends on:** V1 完成
